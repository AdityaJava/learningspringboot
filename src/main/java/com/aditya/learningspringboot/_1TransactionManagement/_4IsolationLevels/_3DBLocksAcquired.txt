# LOCKING BEHAVIOR AT EACH ISOLATION LEVEL
# (Crystal clear, simple words)

====================================================================
1) READ_UNCOMMITTED
====================================================================
READ:
- No shared locks
- Can read uncommitted changes of other transactions

WRITE:
- Writes take exclusive lock until commit

Meaning:
üëâ Reads never lock anything.
üëâ Can see dirty/uncommitted data.

====================================================================
2) READ_COMMITTED
====================================================================
READ:
- Takes shared lock during SELECT
- Releases shared lock immediately after read. That is it doesn't wait for commit or rollback

WRITE:
- Exclusive lock held until commit

Meaning:
üëâ You always see committed data.
üëâ But if you read same row twice, value may change.

====================================================================
3) REPEATABLE_READ
====================================================================
READ:
- Takes shared lock and **keeps it for entire transaction**
- Prevents other transactions from updating the rows you read

WRITE:
- Exclusive lock until commit

(MySQL only)
- Uses "next-key locks" = row lock + gap lock

Meaning:
üëâ If you read a row, nobody can change it until your txn ends.
üëâ In MySQL, also prevents phantom rows using gap locks.

====================================================================
4) SERIALIZABLE
====================================================================
READ:
- Takes range locks / predicate locks
- Prevents inserts/deletes in the range you query

WRITE:
- Exclusive lock until commit

Meaning:
üëâ Highest safety.
üëâ Makes transactions run as if one-by-one.
